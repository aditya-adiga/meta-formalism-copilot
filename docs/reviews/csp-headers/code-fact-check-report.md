## Claim 8: "crypto.randomUUID and Buffer are both available in the Edge runtime that Next proxy runs in."

**Location:** `proxy.ts:35-36`
**Type:** Configuration / Runtime
**Verdict:** Incorrect
**Confidence:** High

The runtime claim is wrong. The bundled Next.js docs explicitly state: "Proxy defaults to using the Node.js runtime. The runtime config option is not available in Proxy files. Setting the runtime config option in Proxy will throw an error." (`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md`, "Runtime" section). The version history confirms the Node.js runtime stabilised in v15.5 and the Middleware → Proxy rename happened in v16.0. `proxy.ts` does not export a `runtime` config, so it runs on the **Node.js runtime**, not the Edge runtime.

The code itself works because `crypto.randomUUID()` and `Buffer` are both available in Node.js 18+ (Buffer is a Node.js global; `crypto.randomUUID()` is on the Web Crypto / Node `crypto` namespace). The behaviour is correct; only the comment's runtime attribution is wrong.

**Evidence:** `proxy.ts:36`, `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md` ("Runtime" section and version-history table)

## Claim 11: "Opt this layout out of static rendering so proxy.ts runs on every request and can attach a fresh per-request CSP nonce."

**Location:** `app/layout.tsx:27-28`
**Type:** Behavioral
**Verdict:** Mostly accurate
**Confidence:** Medium

Calling `await headers()` in a server component does opt that segment into dynamic rendering — confirmed by the Next.js dynamic-APIs documentation pattern. However, the proxy already runs on every matching request regardless of whether `layout.tsx` is dynamic; the proxy and the page-render lifecycle are independent. The actual reason to force dynamic rendering in `layout.tsx` is that **Next.js can only inject the nonce into bootstrap scripts during dynamic rendering** — statically prerendered HTML has no per-request nonce. The Next.js CSP guide explicitly says "your page must be dynamically rendered. ... Next.js applies nonces during server-side rendering, based on the CSP header present in the request. Static pages are generated at build time, when no request or response headers exist—so no nonce can be injected." The comment conflates "proxy.ts runs on every request" (always true for matching paths) with "the layout opts into dynamic rendering" (the actual effect of `await headers()`).

**Evidence:** `app/layout.tsx:31`, `node_modules/next/dist/docs/01-app/02-guides/content-security-policy.md:179-217,385-407`

## Claims Requiring Attention

### Incorrect

- **Claim 8** (`proxy.ts:35-36`): The comment says proxy runs on the Edge runtime; per Next.js 16 docs, Proxy defaults to the Node.js runtime, and setting `runtime` in a proxy file is an error. The code still works because `Buffer` and `crypto.randomUUID()` are also Node.js globals — only the runtime attribution is wrong.

### Mostly Accurate

- **Claim 4** (`proxy.ts:12-14`): "Tailwind v4 emits inline styles" is imprecise. Tailwind v4 builds to external CSS files in this project; the dominant inline-style sources are React `style={{...}}` props in components and Next.js's own SSR style injection. The carve-out itself is justified.
- **Claim 5** (`proxy.ts:16-17`): Anthropic and OpenRouter are server-side as claimed, but OpenAlex code is not present on this branch — the reference is forward-looking. The architectural pattern (server-to-server) does match the existing Anthropic/OpenRouter call sites.
- **Claim 11** (`app/layout.tsx:27-28`): The reason `await headers()` matters is that Next.js requires dynamic rendering to inject the per-request nonce into bootstrap scripts — not that it makes proxy.ts run "on every request" (which it already does for matching paths). Effect is correct; mechanism is misstated.

### Unverifiable

- **Claim 3** (`proxy.ts:9-10`): The "hypothetical injected `<script>`" defense is conditional and not exercised by a specific test or known unsanitized path. Static analysis confirms `react-markdown` sanitizes by default, but proving the negative ("nothing escapes sanitization elsewhere in the app") is out of scope.
- **Claim 9 latent usage** (`proxy.ts:39-40`): The forwarding mechanism is verified but no consumer of `x-nonce` exists in this branch. Whether a future `<Script>` integration would correctly receive the nonce depends on code not yet written.
